# Percept Ingestion & Senses

> Repo: `bud2` | Generated: 2026-04-06 | Commit: 82d9638

## Summary

The percept ingestion pipeline converts raw external signals (Discord messages, calendar events) and internal motivations (impulses) into `Percept` objects that the attention system can score and route. Senses are thin adapters that normalize heterogeneous inputs into `InboxMessage`, which the central `processPercept` function in `main.go` converts into percepts and hands off to the focus queue. The separation â€” sense â†’ `InboxMessage` â†’ `Percept` â€” keeps senses ignorant of memory internals while allowing three distinct signal types (message, signal, impulse) to share one ingestion path.

## Key Data Structures

### `Percept` (`internal/types/types.go`)

The canonical unit of sensory input. Once created, a percept is immutable; all downstream processing (reflex eval, attention scoring, thread linkage) reads from it without mutating it.

Key fields:
- `Source string` â€” origin: `"discord"`, `"github"`, `"calendar"`, or `"impulse:<subtype>"`
- `Type string` â€” event class: `"message"`, `"notification"`, `"event"`, `"due"`, `"wake"`, etc.
- `Intensity float64` â€” pre-computed urgency (0.0â€“1.0); set by the sense, not by attention
- `Data map[string]any` â€” source-specific payload (channel ID, content, event title, etc.)
- `Features map[string]any` â€” clustering hints for thread association
- `DialogueAct string` â€” rule-based classification: `backchannel`, `question`, `command`, `greeting`, `statement`
- `EntropyScore float64` â€” novelty/quality score (0â€“1)
- `ProcessedBy []string` â€” reflex names that handled this percept

### `InboxMessage` (`internal/memory/inbox.go`)

Intermediate message created by senses before percept conversion. The `Type` field gates downstream behavior:

| Type | Subtype examples | Becomes Percept? |
|------|-----------------|-----------------|
| `"message"` (default) | `"thought"` | Yes, via `messageToPercept()` |
| `"impulse"` | `"task"`, `"idea"`, `"system"` | Yes, via `impulseToPercept()` |
| `"signal"` | `"done"` | No â€” control flow only |

### `Impulse` (`internal/types/types.go`)

Internal motivations generated by the system itself (task due, autonomous wake, idea exploration). Separate from external `Percept` but shares the same attention scoring path via `Impulse.ToPercept()`. Stored in `ImpulsePool` separately from `PerceptPool`.

Key fields: `Source ImpulseSource` (task/idea/schedule/system), `Type` (due/upcoming/recurring/explore/wake), `Intensity`, `Data map[string]any`.

### `DiscordSense` (`internal/senses/discord.go`)

Wraps a `*discordgo.Session`. Handles message events, slash commands, connection health monitoring, and hard-reset recovery. Maintains a `pendingInteractions map[string]*PendingInteraction` to correlate deferred slash command responses (tokens expire after 15 minutes).

### `CalendarSense` (`internal/senses/calendar.go`)

Polls Google Calendar every 5 minutes. Tracks state (notified events, daily agenda sent, prediction review sent) in a JSON file at `statePath` so reminders survive restarts. Generates three impulse types: meeting reminders, daily agendas, and sprint review cluster briefs.

### `PerceptPool` (`internal/memory/percepts.go`)

Thread-safe in-memory map of `Percept` objects, backed by a JSON file (`percepts.json`). `ExpireOlderThan()` is called periodically to evict stale entries.

### `ImpulsePool` (`internal/memory/impulses.go`)

Parallel to `PerceptPool` but for internal `Impulse` objects. `Expire()` prunes old impulses; `GetPending()` returns impulses with `intensity > 0` (not yet handled). The pool is JSON-file-backed.

## Lifecycle

1. **Discord WebSocket event**: `discordgo` calls `DiscordSense.handleMessage(s, m)` on an internal discordgo goroutine. The handler filters out self-messages and off-channel messages, classifies the dialogue act (`classifyDialogueAct()`), captures any reply chain, and builds an `InboxMessage{Type: "message"}`.

2. **Slash command path**: `handleInteraction()` receives slash command interactions. `/stop` fires `onStop` synchronously. Other commands send a deferred acknowledgment to Discord (required within 3 seconds), store a `PendingInteraction`, and build an `InboxMessage` with `Extra["slash_command"]` and `Extra["interaction_token"]` for later followup.

3. **Calendar poll loop**: `CalendarSense.pollLoop()` runs on a ticker (default 5 min). Each tick calls `poll()`, which checks for: daily agenda (7â€“9 AM in user timezone), prediction review (14:00 daily), upcoming meetings within reminder window, sprint review clusters. Each produces an `InboxMessage{Type: "impulse"}` sent via `onMessage`.

4. **Direct callback delivery**: Both senses call `onMessage(*InboxMessage)` **synchronously and inline** â€” there is no buffer, channel, or queue between the sense and the callback. This means slow processing in `onMessage` blocks the discordgo event goroutine or the calendar ticker.

5. **processPercept in main.go**: The `onMessage` callback (wired in `cmd/bud/main.go`) calls `msg.ToPercept()`:
   - `"signal"` â†’ returns nil â†’ handled separately as a control signal
   - `"message"` â†’ `messageToPercept()` â†’ builds `Percept` with conversation bucketing and dialogue metadata
   - `"impulse"` â†’ `impulseToPercept()` â†’ builds `Percept` with source `"impulse:<subtype>"`

6. **Conversation ID bucketing** (in `messageToPercept()`): `conversation_id = channel_id + fmt.Sprint(timestamp.Unix()/300)`. Messages in the same channel within a 5-minute window share a conversation ID. This is how threads are associated without explicit thread tracking at the Discord level.

7. **Percept stored**: The resulting `Percept` is added to `PerceptPool.Add()`, persisted to disk. A reference is added to the matching `Thread` via `ThreadPool.AddPerceptRef()`.

8. **Focus queue enqueue**: From the overview, after percept storage, a `PendingItem` is enqueued in `focus.Queue`. The attention system scores it (intensity Ă— source boost Ă— recency) and the highest item drives the next executive wake.

## Design Decisions

- **Inline callback, no buffering**: `onMessage` is called synchronously in the discordgo event handler goroutine. This is a deliberate choice â€” buffering would add latency for urgent inputs and complicate backpressure semantics. The implication is that slow executive processing can briefly stall Discord event delivery.

- **InboxMessage as a normalization layer**: Senses produce `InboxMessage`, not `Percept`. This decouples sense code from memory internals and lets `main.go` own the type-dispatch logic (signal vs. impulse vs. message) in one place.

- **Signal type skips percept creation**: `"signal"` messages (`signal_done`, etc.) return nil from `ToPercept()`. This avoids creating memory artifacts for control-flow events that have no semantic content worth storing or consolidating.

- **Deterministic meeting reminder IDs**: `CalendarSense.sendMeetingReminder()` builds a deterministic `InboxMessage.ID` from the event ID + time window, so duplicate reminders are prevented even if `notifiedEvents` state is lost between restarts.

- **Impulse and Percept coexist**: `ImpulsePool` stores raw `Impulse` objects; when an impulse reaches the executive it has been converted to a `Percept` via `impulseToPercept()`. The original `Impulse` remains in `ImpulsePool` until explicitly `Remove()`d. This allows impulse-level operations (expire, dedup) to remain independent of percept lifecycle.

## Non-Obvious Behaviors

- **`handleMessage` is called on discordgo's internal goroutine**: If `onMessage` blocks (e.g., executive is slow), it backs up Discord event delivery. There is no goroutine spawn or channel buffering here.

- **"thought" subtype gets intensity 0.9**: Bud's own thoughts, generated internally and delivered as Discord messages from the bot itself, are modeled as `InboxMessage{Type: "message", Subtype: "thought"}` and assigned intensity 0.9 â€” higher than most user messages. They go through the full percept pipeline.

- **Calendar only notifies accepted/tentative events**: In `checkUpcomingMeetings()`, events where the user's RSVP status is `declined` or `needsAction` are skipped. Pending invites do not generate reminders.

- **`InboxMessage.Type` defaults to "message" via empty string**: The `Type` field is `omitempty` in JSON serialization. An `InboxMessage` with no explicit type is treated as type "message" in `ToPercept()`. This means any malformed or minimal `InboxMessage` silently goes through the message path.

- **`ThreadPool` resets "active" to "paused" on load**: If the daemon crashes mid-session, the next load finds no active Claude session running. `Load()` finds the "active" thread and resets it to "paused" to prevent the executive from trying to resume a non-existent process.

- **Sprint brief uses keyword clustering, not calendar type**: The sprint review brief logic detects sprint reviews by scanning event titles for keywords (e.g., "sprint review", "sprint planning"), then groups events by local date. If two or more matching events are found on the same day and the first starts within `sprintBriefBefore`, a brief impulse fires. This is purely heuristic â€” there is no calendar category or structured field involved.
